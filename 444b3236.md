## General DBA Foundation

### What daily health checks do you perform?

**Question**  
What daily health checks do you perform as a DBA?

**Strong Answer**  
Daily checks include database availability, service status, CPU, memory, disk usage, I/O behavior, active sessions, long-running queries, blocking sessions, replication health, backup completion, log review, and alert validation. These checks help detect issues before users are impacted.

**Real-world scenario**  
In many environments, a database may appear up while backup failures, replication lag, or high disk growth are silently building risk. Daily health checks reduce this risk.

### What is the difference between backup, restore, and recovery?

**Question**  
Explain backup, restore, and recovery.

**Strong Answer**  
Backup is the process of creating a copy of data. Restore is the process of putting that copy back into a database environment. Recovery means bringing the database to a correct and usable state after restore, often by applying logs or replaying transactions.

**Real-world scenario**  
If a database is lost due to accidental deletion, the backup provides the source, restore rebuilds the environment, and recovery ensures the final state is consistent and usable.

### What are RPO and RTO?

**Question**  
What are RPO and RTO?

**Strong Answer**  
RPO means Recovery Point Objective, which defines how much data loss can be accepted. RTO means Recovery Time Objective, which defines how long the service can remain unavailable before restoration must happen.

**Real-world scenario**  
A critical financial system may have very low RPO and RTO, while a reporting system may tolerate more delay.

### How do you handle a slow production database?

**Question**  
A production database is slow. What do you do first?

**Strong Answer**  
The first step is to confirm the scope of impact and check whether the issue is affecting all users or one application flow. Then review recent changes, active sessions, slow queries, CPU, memory, disk I/O, locks, waits, and database logs. Once the high-impact cause is identified, stabilize the service with the safest action and then fix root cause.

**Real-world scenario**  
A deployment introduces an unoptimized query and user response time increases immediately. A careful check of recent changes and query behavior helps isolate the issue quickly.

## MySQL DBA

### MyISAM vs InnoDB

**Question**  
What is the difference between MyISAM and InnoDB?

**Strong Answer**  
InnoDB supports transactions, row-level locking, crash recovery, and foreign keys, making it suitable for transactional systems. MyISAM is simpler and historically faster for some read-heavy cases but does not support transactions or crash-safe recovery in the same way.

**Real-world scenario**  
An order processing system requires transactional consistency, so InnoDB is the preferred engine.

### What is the InnoDB buffer pool?

**Question**  
What is the InnoDB buffer pool?

**Strong Answer**  
The buffer pool is the memory area where InnoDB caches frequently accessed data pages and index pages. A properly sized buffer pool improves performance by reducing disk reads.

**Real-world scenario**  
If a busy application repeatedly reads the same tables, a well-sized buffer pool helps queries return faster.

### How do you find slow queries?

**Question**  
How do you identify and tune slow queries in MySQL?

**Strong Answer**  
Use the slow query log to identify long-running statements, then check execution plans with `EXPLAIN`. Review whether indexes are missing, whether large table scans are happening, whether joins are inefficient, and whether sorting or grouping is expensive. Based on findings, add indexes, rewrite the query, or adjust workload timing.

**Real-world scenario**  
A reporting query scans a large sales table without using a suitable index, causing CPU spikes during office hours.

### What is replication in MySQL?

**Question**  
What is MySQL replication?

**Strong Answer**  
Replication copies data changes from a primary server to one or more replica servers. It is commonly used for read scaling, standby environments, reporting, and disaster recovery.

**Real-world scenario**  
One primary handles writes while one replica serves read-only reporting traffic.

### How do you troubleshoot replication lag?

**Question**  
What do you do when replication lag becomes high?

**Strong Answer**  
Check whether the lag is caused by large transactions, slow SQL apply on replica, resource shortage, I/O bottlenecks, or network delay. Review replication status, examine resource consumption, and identify whether a specific workload spike caused the delay. After root cause is handled, confirm the replica catches up correctly.

**Real-world scenario**  
A large batch update creates lag for the reporting replica, delaying dashboard freshness.

### What is a deadlock?

**Question**  
What is a deadlock in MySQL?

**Strong Answer**  
A deadlock occurs when two transactions wait on each other’s locked resources and neither can proceed. InnoDB detects this and rolls back one transaction to break the cycle.

**Real-world scenario**  
Two concurrent transactions update the same set of rows in a different order, creating a deadlock.

### Important MySQL commands

- `SHOW PROCESSLIST;`
- `EXPLAIN <query>;`
- `SHOW ENGINE INNODB STATUS\G`
- `SHOW REPLICA STATUS\G`
- `SHOW VARIABLES;`
- `mysqladmin status`

## PostgreSQL DBA

### What is WAL?

**Question**  
What is WAL in PostgreSQL?

**Strong Answer**  
WAL stands for Write-Ahead Logging. PostgreSQL writes changes to WAL before updating the actual data files, which supports crash recovery and replication.

**Real-world scenario**  
If the server crashes unexpectedly, PostgreSQL uses WAL to recover the database to a consistent state.

### Why is VACUUM important?

**Question**  
Why do we need VACUUM in PostgreSQL?

**Strong Answer**  
PostgreSQL uses MVCC, so updates and deletes leave dead tuples behind. VACUUM cleans up these dead tuples, helps control table bloat, and supports healthy database performance.

**Real-world scenario**  
A table with frequent updates grows larger and slower over time if vacuuming is not working properly.

### What is autovacuum?

**Question**  
What is autovacuum?

**Strong Answer**  
Autovacuum is the automatic background process that runs vacuum and analyze based on table activity. It prevents excessive bloat and keeps planner statistics updated.

**Real-world scenario**  
Without autovacuum, frequently updated tables can become inefficient and cause performance issues.

### How do you tune a slow PostgreSQL query?

**Question**  
How do you troubleshoot a slow PostgreSQL query?

**Strong Answer**  
Use `EXPLAIN ANALYZE` to inspect the actual execution plan. Review sequential scans, join strategies, sorts, row estimates, and index usage. Then improve the query, add or adjust indexes, and ensure statistics are up to date.

**Real-world scenario**  
A query becomes slow because stale statistics lead PostgreSQL to choose a poor execution plan.

### How do you find blocking sessions?

**Question**  
How do you handle blocking in PostgreSQL?

**Strong Answer**  
Check `pg_stat_activity` and lock information to identify the blocking session and blocked sessions. Understand whether the blocking transaction is valid or stuck, then decide whether to wait, cancel, or terminate it based on business impact and safety.

**Real-world scenario**  
A transaction is left open for too long and blocks updates from multiple users.

### What is streaming replication?

**Question**  
What is streaming replication?

**Strong Answer**  
Streaming replication continuously transfers WAL records from primary to standby. It is used to maintain a synchronized standby server for high availability or read access.

**Real-world scenario**  
If the primary server fails, the standby can be promoted to reduce downtime.

### Important PostgreSQL commands

- `SELECT * FROM pg_stat_activity;`
- `SELECT * FROM pg_stat_replication;`
- `EXPLAIN ANALYZE <query>;`
- `VACUUM ANALYZE <table>;`
- `\l`
- `\du`

## MongoDB DBA

### What is MongoDB?

**Question**  
What is MongoDB?

**Strong Answer**  
MongoDB is a document-oriented NoSQL database that stores data in BSON documents. It is useful for applications with flexible or evolving data structures.

**Real-world scenario**  
An application with different product attributes for different categories uses MongoDB because schema flexibility is helpful.

### What is a replica set?

**Question**  
What is a replica set in MongoDB?

**Strong Answer**  
A replica set is a group of MongoDB nodes where one primary accepts writes and secondary nodes replicate the data. It provides redundancy and automatic failover.

**Real-world scenario**  
If the primary node becomes unavailable, a secondary can be elected and the application continues with less disruption.

### What is sharding?

**Question**  
What is sharding in MongoDB?

**Strong Answer**  
Sharding distributes data across multiple servers so that storage and workload can scale horizontally. It is used when a single server is no longer enough for the data size or throughput.

**Real-world scenario**  
An analytics platform handling very large event data may shard collections to spread load.

### How do indexes help in MongoDB?

**Question**  
How do indexes improve MongoDB performance?

**Strong Answer**  
Indexes reduce the number of documents scanned and improve query speed. They are especially useful for filtering, sorting, and lookup operations, but too many indexes can increase write cost and storage usage.

**Real-world scenario**  
A query filtering by customer ID becomes faster after the correct index is added.

### How do you find slow queries in MongoDB?

**Question**  
How do you identify slow operations in MongoDB?

**Strong Answer**  
Use the profiler, query analysis tools, and execution statistics to find slow operations. Review filter fields, sort patterns, index usage, and whether the document model is causing unnecessary complexity.

**Real-world scenario**  
A dashboard query scans too many documents because the right compound index is missing.

### Important MongoDB commands

- `show dbs`
- `show collections`
- `db.serverStatus()`
- `rs.status()`
- `db.currentOp()`
- `db.collection.createIndex()`

## Backup And Recovery

### MySQL backup answer

**Question**  
How do you take backup and restore in MySQL?

**Strong Answer**  
Backup can be taken using logical or physical methods depending on data size and recovery goals. Restore should first be tested in a controlled environment, then validated for schema, row count, consistency, and application readiness. If point-in-time recovery is needed, binary logs are applied after restore.

### PostgreSQL backup answer

**Question**  
How do you handle backup and recovery in PostgreSQL?

**Strong Answer**  
PostgreSQL supports logical backups like `pg_dump` and physical backups for full cluster recovery. For point-in-time recovery, a base backup is combined with archived WAL files so the database can be restored to an exact point before the failure.

### MongoDB backup answer

**Question**  
How do you take backup and restore in MongoDB?

**Strong Answer**  
MongoDB backup can be done using logical tools like `mongodump` and `mongorestore`, or with snapshot-based methods for large production setups. The chosen approach depends on scale, speed, and restore expectations.

### Real-world restore principle

A backup is only useful when restore has been tested. Many interviewers expect this point because backup success alone does not prove recovery readiness.

## Security And Access

### How do you manage database access?

**Question**  
How do you manage user access securely?

**Strong Answer**  
Access should follow the principle of least privilege. Users receive only the permissions needed for their tasks, and access is reviewed regularly. Sensitive administrative access is controlled, audited, and restricted to approved personnel.

**Real-world scenario**  
A reporting user needs read-only access to one schema, not full database-level permissions.

### What is the least privilege principle?

**Question**  
What is least privilege?

**Strong Answer**  
Least privilege means giving the minimum level of access required to perform a task. It reduces security risk and prevents accidental damage.

## Scenario-Based Interview Questions

### Scenario 1: Production slowness after deployment

**Question**  
Users say the application is slow just after a new release. What do you do?

**Strong Answer**  
First confirm the time the issue started and compare it with the deployment window. Review recent changes, database logs, active queries, locks, CPU, memory, and slow query indicators. If the issue is linked to a new query pattern or missing index, stabilize by rollback or immediate tuning, then monitor closely.

**Real-world insight**  
This scenario tests whether troubleshooting starts from evidence and recent changes, not guesswork.

### Scenario 2: Replication stopped

**Question**  
Replication has stopped. What is your approach?

**Strong Answer**  
Check replication error details first, not just service status. Review logs, network health, disk space, credentials, incompatible changes, and whether transactions are blocked or broken. Fix the exact cause, restart carefully, and confirm replication catches up without data inconsistency.

### Scenario 3: Disk usage reached critical level

**Question**  
The database server disk is 95 percent full. What will you do?

**Strong Answer**  
Immediately identify what is consuming the space such as logs, archived files, temporary files, or data growth. Free space safely without deleting critical files blindly, stabilize the service, then implement retention, alerting, and capacity planning improvements.

**Real-world insight**  
This tests operational maturity. Unsafe file deletion can create bigger outages.

### Scenario 4: Accidental data deletion

**Question**  
A critical table was dropped accidentally. How do you recover it?

**Strong Answer**  
Assess available backups, recovery point requirements, and business urgency. Restore to a separate environment first, recover the required data set, validate it, and then reintroduce it safely into production. Communicate clearly during the process.

### Scenario 5: Blocking issue

**Question**  
One long transaction is blocking other sessions. What do you do?

**Strong Answer**  
Identify the blocking session, confirm whether it is active, stuck, or waiting for application action, and understand the business impact. If required, cancel or terminate it carefully after coordination, then address the root cause such as poor transaction handling or long-running application logic.

### Scenario 6: Backup passed but restore failed

**Question**  
Backup completed successfully, but restore failed. What does that mean?

**Strong Answer**  
It means backup completion alone did not confirm recovery readiness. The process must include restore testing, integrity validation, and recovery drills. A valid DBA process always tests recovery, not just backup execution.
